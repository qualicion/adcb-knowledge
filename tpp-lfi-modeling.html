1. Mission & Scope

Verify that the ADCB LFI (Licensed Financial Institution) da-of-api codebase adequately protects against Open Finance liabilities as defined by CBUAE regulations. The system handles:

  • Data Sharing - Accounts, transactions, balances, beneficiaries, customer data, standing orders, direct debits
  • Payment Processing - Payment Hub integration, PII decryption (JWE/JWT), creditor/debtor mapping, refund handling
  • Authorization - Cognito authentication, JWE decoding, session management, OIDC flows
  • Consent Management - Dynamic scope validation, boundary enforcement, usage limits
  • Security - Ecosystem boundary validation, fraud detection (anomaly + VRP/SCA), data leak prevention
  • Liability Mitigation - Transaction frequency/amount checks, geographic restrictions, data transmission controls

SFDPOT Heuristic Decomposition

James Bach's SFDPOT Quality Criteria Applied to LFI

Each dimension surfaces unique test ideas that pure requirements-based testing would miss.

S - Structure (What the product IS)

  • Code modules: 3 Lambda functions (authorizer, data_sharing, payment) - but consent & security modules are not wired into Lambda handlers
  • Architecture: AWS Lambda + API Gateway + DynamoDB + Secrets Manager
  • Dependencies: jwt, jwcrypto, requests, shortuuid, aws-lambda-powertools - no dependency pinning in fraud modules
  • File layout gap: src/security/fraud/ directory doesn't exist despite IMPLEMENTATION_GUIDE.md references
  • Test coverage: 80+ unit tests exist for data_sharing/authorizer/payment - 0 tests for consent/ and security/ modules
  • Data mapping: 12 mapping modules (accounts, balances, transactions, etc.) - all need schema validation tests

F - Function (What the product DOES)

  • Payment flow: PII decryption - payload mapping - Payment Hub POST - response mapping - Ozone format. Test each transformation.
  • Data sharing: Account/balance/transaction retrieval from CDP/CRM sources with field mapping. Validate mapping accuracy.
  • Authorization: Cognito auth - JWE decode - Procash/Ozone proxy validation - session management. Test token expiry edge cases.
  • Consent validation: Scope checking, expiry, usage limits, geographic/IP/device restrictions. All use mock data - untestable in production.
  • Boundary enforcement: Account-level, field-level, quota-level checks with audit trails. Logic exists but no integration points.
  • Fraud detection: Frequency, amount, timing anomaly detection. Referenced but not implemented in codebase.

D - Data (What the product PROCESSES)

  • PII data: JWE-encrypted personal info (creditor names, IBANs). Test with Arabic/bilingual names, special characters.
  • Payment amounts: AED currency with decimal precision. Test boundary at AED 50,000 threshold.
  • Account IDs: Surrogate keys mapping to real accounts. Test surrogate-to-real mapping integrity.
  • Transaction history: Up to 365 days. Test at 90-day and 365-day boundaries.
  • Consent scopes: accounts, transactions, balances, beneficiaries, customer, products, standing_orders, direct_debits, scheduled_payments.
  • Response mapping: PaymentHub PascalCase to Ozone camelCase. Test key conversion edge cases.
  • Charge bearer values: BorneByCreditor to DEBT, BorneByDebtor to CRDT, FollowingServiceLevel to SLEV, Shared to SHAR. Test unknown values (returns None).

P - Platform (What the product DEPENDS ON)

  • AWS Lambda: Cold start latency vs 30-second Payment Hub timeout. Test timeout cascade.
  • Payment Hub: External API with Basic Auth. Test when credentials rotate, SSL cert changes, 5xx responses.
  • DynamoDB: Consent storage, fraud monitoring tables. Test eventual consistency race conditions.
  • Secrets Manager: Payment Hub credentials, JWK keys. Test rotation and cache invalidation.
  • API Gateway: Request routing, throttling, WAF. Test request size limits vs data transmission limits.
  • Network: root_ca_bundle_path for SSL verification. Test with expired/invalid CA bundles.
  • JWK cache: lru_cache(maxsize=None) for JWKS clients - unbounded cache is a memory leak risk.

O - Operations (HOW it will be used)

  • TPP integration: Multiple TPPs hitting APIs concurrently. Test concurrent consent validations.
  • Monitoring: CloudWatch metrics referenced but not implemented. No fraud alerts in production.
  • Incident response: What happens when fraud is detected? No escalation workflow exists.
  • Deployment: Canary deployment mentioned but no feature flags in codebase.
  • Error handling: Payment Hub errors map to generic error codes. Test error code differentiation.
  • Audit trail: Boundary violations logged to in-memory list (self.violation_log). Lost on Lambda cold start.
  • Configuration: Risk thresholds hardcoded. No environment variable override mechanism.

T - Time (WHEN things happen)

  • Consent expiry: 90-day default. Test at expiry boundary (second before/after). Test timezone handling (UTC only).
  • Token lifecycle: JWT leeway of 10 seconds. Test with tokens at exactly 10s past expiry.
  • Rate limiting: 5 concurrent requests/hour for consent, 60 API calls/minute. Test at exact threshold.
  • Restricted hours: 22:00-06:00 UTC for sensitive scopes. Test at boundary (21:59, 22:00, 06:00, 06:01).
  • Transaction frequency: 50/day limit. Test at 49, 50, 51 transactions.
  • Daily transmission reset: Uses UTC date string. Test around midnight UTC.
  • Payment Hub timeout: 30 seconds. Test at 29s, 30s, 31s response times.

👤 You — the Customer ✕
You are the end customer — a real person with a bank account. You use a TPP app and give consent for it to read your data or make payments on your behalf. You stay in control at all times and can revoke consent whenever you want. UAE law guarantees this right.
📱 TPP — Third Party Provider ✕
TPP = Third Party Provider. The app you are actually using — a budgeting tool, merchant checkout, P2P app, or finance aggregator. Licensed by CBUAE (minimum AED 1M capital). It calls Open Finance APIs on your behalf but never sees your bank password. Must hold FAPI 2.0 certification.

🔀 API Hub — Al Tareq / Nebras ✕
The API Hub is the central nervous system of UAE Open Finance. Built on Ozone technology, operated by Nebras, branded as Al Tareq for consumers. It handles: (1) Security — OAuth 2.0, FAPI 2.0, mTLS certificates. (2) Consent management — creating, tracking, and revoking consents. (3) Routing — directing every API call between TPPs and LFIs. Think of it as air traffic control — every single interaction flows through here.

The complete story of consent and data — told simply, shown clearly

Every system, every step, every data movement — explained in plain language with the technical terms woven in, so you understand both what is happening and why it matters.

The one idea that unlocks everything

This entire system exists to answer one question: can our platform access a user's bank data — safely, legally, and without storing a password? The answer is yes, and it works like a chain of sealed envelopes and locked vaults, each one only opened by the right key. Once you see that image, every step below will click immediately.

Part 1

Getting permission — the consent journey

A user has a bank account. We want to read it on their behalf. We cannot just ask for their password — that would be insecure and illegal. Instead, we go through a formal handshake involving signed letters, one-time vouchers, and encrypted vaults. Here is every step, in order.

What each step does — in plain words

Step 1 — The ID card: Our app proves who it is

Picture a visitor arriving at a high-security building. They go to the front desk, show their ID, and get a timed visitor badge. Our app does exactly this — it calls AWS Cognito and provides its client_id and client_secret. Cognito hands back a JWT (JSON Web Token) — a digitally sealed envelope that says "this is who I am, and here is what I am allowed to do." The envelope expires in minutes so nobody can steal and reuse it.

Technical note: This is called the client_credentials grant — machine to machine, no user involved yet. The JWT contains scopes (permissions). For example, write/consents means "this app is allowed to start a consent." Without the right scope, nothing works.

Steps 2, 3, 4 — The bouncer: Kong checks every request

Think of Kong Gateway as airport security. Every request must pass through it — no exceptions. Three checks in under a second:

  • Check 1 — Is the signature real? Kong verifies the JWT against a public key list called JWKS. Only Cognito's fingerprints are on the list.
  • Check 2 — Has it expired? JWT tokens go stale fast. An old one is rejected immediately.
  • Check 3 — Right permission? Each route needs a specific scope. No scope = no entry.

Once all three pass, Kong strips the JWT and stamps a clean header called X-Client-Id. The TPP backend never sees a raw security token.

Test signal: If you ever see an Authorization header reaching the TPP backend — something is wrong with Kong. That header must be stripped every single time, without fail.

Step 5 — Opening the file: TPP creates the consent record

A detective opens a new case file and locks away the evidence. A new row is written to DynamoDB (tpp_consents table). Status: AwaitingAuthorization. Two secret values are generated immediately:

  • PKCE verifier — a random key that proves our callback is genuine and was not tampered with. Think of it as a sealed wax stamp only we can reproduce.
  • oauth_state — a unique ticket number that prevents anyone from forging a fake callback. This is called CSRF protection.

Both are encrypted with AES-256-GCM before saving — not just hashed, fully encrypted. Even database access gives you nothing without the key.

Steps 6, 7 — The signed letter: TPP sends a formal request to the bank

The TPP cannot reach the user's bank directly. Every external call goes through the OF API Hub — the UAE's official Open Finance routing centre, like a certified postal service that only delivers between licensed institutions.

The letter is a JAR (JSON Authorization Request), signed with PS256 — a wax seal mathematically impossible to forge. It contains the exact permissions, packaged as a RAR (Rich Authorization Request), and travels over mTLS — a tunnel where both sides show identity certificates. The bank replies with the authorization_url we send to the user.

Steps 8, 9 — The user decides: Bank callback and verification

The bank shows the user exactly what they are agreeing to share. The user clicks Allow. The bank sends back a one-time authorization_code (expires the moment it is used) and the state value. Our first action is to look up that state via the oauth_state-index.

CSRF check: If the state does not match — we stop immediately. No token exchange, no data, nothing. An attacker who tried to forge a callback would not know our state value — so they cannot get through.

Step 10 — The vault: We swap the voucher for three encrypted keys

The TPP decrypts the PKCE verifier, combines it with the authorization_code, and calls the bank's token endpoint via the OF API Hub over mTLS. The bank verifies everything matches the original signed letter, and hands back three tokens:

access_token~1 hour TTL
refresh_tokenLong-lived
id_tokenReference ID

The actual key used to call the bank's API. Goes in every data request.

When the access_token expires, we quietly use this to get a new one — no user action needed.

Contains the bank's own consent reference (of_consent_id). Proves which consent we are acting on.

Consent status flips: AwaitingAuthorization → Authorized. An entry is written to tpp_audit_log — permanent, append-only. Part 1 is complete. The user did their part once. Everything from here is automatic.

Part 2

The data fetch — the night shift that never stops

The user gave permission once and walked away. From here, a background worker runs automatically on a schedule — silently fetching the latest financial data, checking tokens, handling failures, and shipping everything to our data platform. No human involvement. No data stored in our database.

Steps 1-2 — The night shift worker

An EventBridge rule fires on schedule — like an alarm clock. The TPP backend wakes up and queries the tpp_consents table: "Give me every consent where status is Authorized and expiration_date has not passed." Each consent is processed independently — a problem with one does not delay the others.

Steps 3-4 — Checking the key before the call

The scheduler gets the most recent token via ScanIndexForward=False, Limit=1. If it will expire within TOKEN_REFRESH_BUFFER_SECONDS, it renews it now — before the API call. The new token is encrypted and saved. The old one stays for audit purposes.

401 on refresh = the user revoked access at their bank. Consent is immediately marked Revoked, revoked_by = system. No retry. No exceptions.

Step 5 — Three calls, one secure tunnel

Before calling the bank, the TPP generates a fresh x-fapi-interaction-id — a unique UUID required under the FAPI standard. Three separate API calls through the OF API Hub over mTLS:

Step 6 — The conveyor belt: Data lands on Kafka

The TPP enriches the data — adds consent_id, channel_id, lfi_id, and fetch timestamp — then publishes to EDP Kafka. The TPP's job ends the moment the message is on the belt.

Golden rule: Scan all seven DynamoDB tables after a successful fetch. You will find zero account numbers, zero balances, zero transactions. Only consent metadata, encrypted tokens, and the audit log live in DynamoDB.

The right response depends entirely on what is actually wrong — not how loud the alarm is.

The complete story of consent and data — told simply, shown clearly

Every system, every step, every data movement — explained in plain language with the technical terms woven in, so you understand both what is happening and why it matters.

The one idea that unlocks everything

This entire system exists to answer one question: can our platform access a user's bank data — safely, legally, and without storing a password? The answer is yes, and it works like a chain of sealed envelopes and locked vaults, each one only opened by the right key. Once you see that image, every step below will click immediately.

Getting permission — the consent journey

A user has a bank account. We want to read it on their behalf. We cannot just ask for their password — that would be insecure and illegal. Instead, we go through a formal handshake involving signed letters, one-time vouchers, and encrypted vaults. Here is every step, in order.

What each step does — in plain words

Step 1 — The ID card: Our app proves who it is

Picture a visitor arriving at a high-security building. They go to the front desk, show their ID, and get a timed visitor badge. Our app does exactly this — it calls AWS Cognito and provides its client_id and client_secret. Cognito hands back a JWT (JSON Web Token) — a digitally sealed envelope that says "this is who I am, and here is what I am allowed to do." The envelope expires in minutes so nobody can steal and reuse it.

Technical note: This is called the client_credentials grant — machine to machine, no user involved yet. The JWT contains scopes (permissions). For example, write/consents means "this app is allowed to start a consent." Without the right scope, nothing works.

Steps 2, 3, 4 — The bouncer: Kong checks every request

Think of Kong Gateway as airport security. Every request must pass through it — no exceptions. Three checks in under a second:

  • Check 1 — Is the signature real? Kong verifies the JWT against a public key list called JWKS. Only Cognito's fingerprints are on the list.
  • Check 2 — Has it expired? JWT tokens go stale fast. An old one is rejected immediately.
  • Check 3 — Right permission? Each route needs a specific scope. No scope = no entry.

Once all three pass, Kong strips the JWT and stamps a clean header called X-Client-Id. The TPP backend never sees a raw security token.

Test signal: If you ever see an Authorization header reaching the TPP backend — something is wrong with Kong. That header must be stripped every single time, without fail.

Step 5 — Opening the file: TPP creates the consent record

A detective opens a new case file and locks away the evidence. A new row is written to DynamoDB (tpp_consents table). Status: AwaitingAuthorization. Two secret values are generated immediately:

  • PKCE verifier — a random key that proves our callback is genuine and was not tampered with. Think of it as a sealed wax stamp only we can reproduce.
  • oauth_state — a unique ticket number that prevents anyone from forging a fake callback. This is called CSRF protection.

Both are encrypted with AES-256-GCM before saving — not just hashed, fully encrypted. Even database access gives you nothing without the key.

Steps 6, 7 — The signed letter: TPP sends a formal request to the bank

The TPP cannot reach the user's bank directly. Every external call goes through the OF API Hub — the UAE's official Open Finance routing centre, like a certified postal service that only delivers between licensed institutions.

The letter is a JAR (JSON Authorization Request), signed with PS256 — a wax seal mathematically impossible to forge. It contains the exact permissions, packaged as a RAR (Rich Authorization Request), and travels over mTLS — a tunnel where both sides show identity certificates. The bank replies with the authorization_url we send to the user.

Steps 8, 9 — The user decides: Bank callback and verification

The bank shows the user exactly what they are agreeing to share. The user clicks Allow. The bank sends back a one-time authorization_code (expires the moment it is used) and the state value. Our first action is to look up that state via the oauth_state-index.

CSRF check: If the state does not match — we stop immediately. No token exchange, no data, nothing. An attacker who tried to forge a callback would not know our state value — so they cannot get through.

Step 10 — The vault: We swap the voucher for three encrypted keys

The TPP decrypts the PKCE verifier, combines it with the authorization_code, and calls the bank's token endpoint via the OF API Hub over mTLS. The bank verifies everything matches the original signed letter, and hands back three tokens:

access_token~1 hour TTL
refresh_tokenLong-lived
id_tokenReference ID

The actual key used to call the bank's API. Goes in every data request.

When the access_token expires, we quietly use this to get a new one — no user action needed.

Contains the bank's own consent reference (of_consent_id). Proves which consent we are acting on.

Consent status flips: AwaitingAuthorization → Authorized. An entry is written to tpp_audit_log — permanent, append-only. Part 1 is complete. The user did their part once. Everything from here is automatic.

The data fetch — the night shift that never stops

The user gave permission once and walked away. From here, a background worker runs automatically on a schedule — silently fetching the latest financial data, checking tokens, handling failures, and shipping everything to our data platform. No human involvement. No data stored in our database.

Steps 1-2 — The night shift worker

An EventBridge rule fires on schedule — like an alarm clock. The TPP backend wakes up and queries the tpp_consents table: "Give me every consent where status is Authorized and expiration_date has not passed." Each consent is processed independently — a problem with one does not delay the others.

Steps 3-4 — Checking the key before the call

The scheduler gets the most recent token via ScanIndexForward=False, Limit=1. If it will expire within TOKEN_REFRESH_BUFFER_SECONDS, it renews it now — before the API call. The new token is encrypted and saved. The old one stays for audit purposes.

401 on refresh = the user revoked access at their bank. Consent is immediately marked Revoked, revoked_by = system. No retry. No exceptions.

Step 5 — Three calls, one secure tunnel

Before calling the bank, the TPP generates a fresh x-fapi-interaction-id — a unique UUID required under the FAPI standard. Three separate API calls through the OF API Hub over mTLS:

Step 6 — The conveyor belt: Data lands on Kafka

The TPP enriches the data — adds consent_id, channel_id, lfi_id, and fetch timestamp — then publishes to EDP Kafka. The TPP's job ends the moment the message is on the belt.

Golden rule: Scan all seven DynamoDB tables after a successful fetch. You will find zero account numbers, zero balances, zero transactions. Only consent metadata, encrypted tokens, and the audit log live in DynamoDB.

Liability Validations & Compliance.
What They Mean, How They Work.

ADCB is the bank — the Licensed Financial Institution. When a TPP asks for data or sends a payment, ADCB is the one fulfilling that request. If something goes wrong — data leaks, payments fail, consent is ignored — CBUAE holds ADCB financially liable. This section explains every liability, how the codebase protects against it, and where the gaps are.

The one idea that unlocks everything here

Think of ADCB as a vault operator. TPPs are authorised visitors who can open specific deposit boxes — but only the ones the customer said they could, only during visiting hours, and only if they show the right ID every time. The liability framework is the penalty schedule for when ADCB leaves a box open, lets the wrong visitor in, or loses track of who accessed what. Every AED fine below maps to a specific failure in that vault metaphor.

Liability Framework

What the fines mean — and why they exist

CBUAE's Open Finance Framework defines specific AED penalties for every type of failure. These are not theoretical — they are contractual obligations between the LFI, TPPs, and the regulator. The fines are split into two sides: LFI liabilities (when the bank makes a mistake) and TPP liabilities (when the fintech app makes a mistake).

Key rules to remember

  • Indirect, consequential, and tortious losses are not compensated under OF standards — only direct losses.
  • Nebras (API Hub) liability for Trust Framework failures is capped at AED 5,000,000 per claim.
  • Users bear liability only for errors they directly supply — e.g. incorrect beneficiary details they typed themselves.
  • Fines are per incident — a single bug that affects 100 customers means 100 separate liabilities.

Reading the tables above

The highest fines — AED 5,000 — are reserved for two scenarios: destroying a TPP's long-lived consent through a data mapping bug, and breaking API changes without notice. These are the "nuclear" penalties because they affect the entire ecosystem, not just one customer. The most common fines — AED 250–500 — cover accuracy and fraud monitoring failures. These happen per-incident, so a single bug affecting 1,000 customers means up to AED 500,000 in exposure.

How it works

The validation layers that prevent liability

The LFI codebase (da-of-api) is designed around six defensive layers. Each layer maps directly to a category of liability in the tables above. When a request arrives from a TPP, it should pass through every relevant layer before data is returned or a payment is executed.

Think of it as airport security

Every TPP request goes through multiple checkpoints. Layer 1 checks your passport (authentication). Layer 2 checks your boarding pass matches your destination (consent). Layer 3 checks you're not carrying prohibited items (boundary enforcement). Layer 4 checks you're not on a watchlist (fraud detection). Layer 5 makes sure your luggage goes to the right plane (data mapping). Layer 6 tracks everything on camera (audit trail). Skip a checkpoint and something gets through that shouldn't.

Layer 1 — Authentication (Protects against AED 500)

Before anything else, prove you are who you claim to be. The authoriser Lambda validates every incoming request.

Authentication flow — three paths
CognitoJWT token auth
›
JWE DecodePS256 signature
›
JWKS VerifyPublic key lookup
›
Authorised10s leeway

How it works: The authoriser checks three auth paths — Cognito (internal), Procash (banking), and Ozone proxy (TPP). Every JWT is verified against the issuer's JWKS endpoint using PS256 signatures. Tokens get a 10-second leeway window — a token that expired 9 seconds ago is still accepted, but at 11 seconds it's rejected. This prevents clock-skew failures without opening a security hole.

Liability it prevents: AED 500 for "Fraudulent/erroneous LFI authentication". If the JWT verification is bypassed, an attacker could impersonate any customer.

Layer 2 — Consent Validation (Protects against AED 350–5,000)

The customer said the TPP can see their accounts and balances. Did they also say transactions? Did the consent expire? Has the TPP already used it 1,000 times today?

Consent validation — five checks
Scope MatchGranted vs requested
›
Expiry Check90-day window
›
Usage CountMax 1,000 calls
›
Time Window22:00–06:00 block
›
Enhanced AuthSensitive scopes

How it works: The DynamicScopeValidator checks five things in order:

  • Scope match — is the requested data cluster (accounts, transactions, beneficiaries) in the list the customer approved?
  • Expiry — has the 90-day consent window passed? Checked to the second, in UTC.
  • Usage limits — has this consent been used more than 1,000 times? Prevents data harvesting.
  • Time restrictions — sensitive scopes (customer PII, beneficiaries) are blocked between 22:00–06:00 UTC. If someone breaches data at 3am, response time is slower = bigger damage.
  • Enhanced authentication — restricted fields (national ID, date of birth) require step-up authentication beyond the basic token.

Liability it prevents: AED 350 (expired consent), AED 750 (data outside scope), AED 5,000 (consent destruction through mapping failure).

Layer 3 — Boundary Enforcement (Protects against AED 750–5,000)

Even with valid consent, not everything should be visible. Boundary enforcement controls what data leaves, where it goes, and how much at a time.

Three types of boundaries:

  • Account boundaries — The TPP can only see accounts listed in the consent. Request acc_3 (the customer's mortgage) when consent only covers acc_1 and acc_2? Blocked. Action: BLOCK.
  • Field boundaries — Sensitive fields (IBAN, email, phone) are masked before leaving ADCB. Full IBAN AE07 0331 2345 6789 0123 456 becomes **********************3456. National ID and date of birth are classified as restricted and require enhanced auth. Action: ESCALATE.
  • Quota boundaries — Maximum 100 records per request, 10MB per transmission. A TPP trying to download 500 transactions in one call gets capped with pagination. Action: FILTER.

Ecosystem boundary validation goes further — it checks where the data is going. Is the destination an authorised ecosystem participant (ADCB internal, CBUAE regulatory, Ozone partner)? Is the destination country in the GCC allowed list? Is the source-to-destination route crossing a restricted border (e.g. UAE to Iran)?

The EcosystemBoundaryValidator classifies every outbound transmission and blocks anything that violates geographic restrictions, data type rules, or transmission size limits.

Layer 4 — Fraud Detection (Protects against AED 250–500)

Is this payment normal? Is this access pattern suspicious? Fraud detection looks at three dimensions.

Critical gap: this layer is not yet built

The Implementation Guide references anomaly_detector.py, vrp_sca_monitor.py, and fraud_integration_handler.py — but none of these files exist in the codebase. The src/security/fraud/ directory is empty. All fraud detection claims are documentation-only.

What should exist:

  • Frequency detection — 50 transactions/day limit per customer. The 51st should be blocked.
  • Amount anomaly — a customer who normally spends AED 200–500 suddenly sends AED 49,999. Pattern analysis should flag this.
  • VRP monitoring — Variable Recurring Payments need their own fraud checks. A merchant charging AED 950 twelve times in one day against a "max AED 1,000/month" VRP should be caught.
  • Geographic restrictions — payments to sanctioned countries (Syria, Iran, North Korea) must be blocked immediately.

Layer 5 — Data Mapping & Payment Execution (Protects against AED 250–500)

Once a request passes all checks, it's time to actually fetch the data or send the payment. This is where field mapping accuracy matters.

Data sharing path: The data_sharing Lambda fetches from CDP (Customer Data Platform) or CRM, maps the response through one of 12 mapping modules (accounts, balances, transactions, beneficiaries, customer, products, standing orders, direct debits, scheduled payments, offers, parties, statements), and returns it in Ozone-compliant format.

Payment path: The payment Lambda decrypts the PII section (JWE-encrypted creditor name, IBAN), maps it from Ozone format to Payment Hub format (camelCase → PascalCase), sends it to the Payment Hub with a 30-second timeout, and maps the response back. Key mapping: BorneByCreditor→DEBT, BorneByDebtor→CRDT, FollowingServiceLevel→SLEV, Shared→SHAR.

Known regression: payment status check is disabled

At post_payments.py:657–664, the transaction status validation is commented out. This means Payment Hub can return HTTP 201 (looks successful) with TransactionStatus = "RJCT" (actually rejected), and the customer is told their payment succeeded when it didn't. This is an AED 350 liability per occurrence — "Failure to execute request accurately".

Layer 6 — Audit Trail (Supports all liability categories)

Every decision — allow, block, escalate — must be recorded. When CBUAE audits ADCB, they will ask: "show me every time you blocked a request, and show me every time you allowed one." Without a complete audit trail, ADCB cannot defend against any liability claim.

Known weakness: audit data is stored in memory

The boundary violation_log, fraud customer_activity, and ecosystem transmission_log are all stored as in-memory Python lists/dicts. When the Lambda cold-starts (AWS scales down and back up), all audit data is lost. A violation at 09:00 won't appear in the log at 09:05 if the Lambda recycled in between. This makes audit trail completeness impossible to guarantee.

Fix needed: Persist all audit data to DynamoDB or CloudWatch, not in-memory structures.

Test Strategy

Rapid Software Testing — liability-driven test design

This test strategy uses James Bach's RST (Rapid Software Testing) methodology. Every test case maps to a specific AED liability. Higher fines get more test depth. The approach uses SFDPOT heuristics (Structure, Function, Data, Platform, Operations, Time) and consistency oracles to find gaps that requirements-based testing would miss.

How we judge test results — six oracles

An oracle is how you decide if the system behaved correctly. We use six:

  • Consistency Oracle — does behavior match CBUAE OF specs, Ozone API contract, and ADCB business rules?
  • History Oracle — does the current version behave consistently with prior releases? Are regressions caught?
  • Image Oracle — does the system project the trustworthiness expected of a regulated bank?
  • Comparable Product Oracle — how does it compare with UK Open Banking (OBIE) and EU PSD2?
  • Claims Oracle — does the codebase deliver what the Implementation Guide claims?
  • Reference Oracle — does it comply with CBUAE penalties, ISO 27001, and data protection requirements?

Test Automation — the pyramid

Of the 156 test cases identified, roughly 85% can be automated. The remaining 15% need manual exploratory testing (UX consent flows, accessibility, security pen testing). Here's how the test pyramid breaks down:

Unit tests (96 tests) — absolutely essential

Every consent validator, boundary enforcer, risk calculator, and data mapper is pure Python logic with zero external dependencies. These are the cheapest, fastest tests to write and run. Skipping them would be negligent for a regulated system.

  • Consent scope validation — 20 tests covering scope matching, expiry boundaries (1 second before/after), usage counts, restricted hours, hierarchical scopes
  • Boundary enforcement — 15 tests covering account authorisation, field masking, quota limits, data type restrictions
  • Ecosystem boundary — 12 tests covering cross-border validation, PII classification, transmission limits, restricted countries
  • Payment mapping — 14 tests covering creditor name dicts, charge bearer mapping, PascalCase→camelCase conversion, Arabic UTF-8 preservation
  • Data mapping modules — 18 tests covering all 12 mapping files for mandatory fields, address assembly, balance precision

Integration tests (45 tests) — necessary, with a caveat

The biggest risk isn't that individual functions are wrong — it's that the modules aren't connected. Consent validators exist but Lambda handlers don't call them. Integration tests verify the wiring.

  • Lambda → Consent Validator — 8 tests. The critical missing wiring. Write the test first (red), then wire the code (green).
  • Lambda → Boundary Enforcer — 6 tests. Account filtering, field masking, violation persistence (not in-memory).
  • Payment → Payment Hub — 8 tests. Timeout handling, status code validation, SSL failure recovery.
  • Consent → DynamoDB — 5 tests. Replace all mock _get_consent_data() with real lookups using moto.

Caveat: Integration tests for consent/boundary wiring can't be written until those modules are actually imported into Lambda handlers. Write the test first — this is TDD at the integration level.

E2E tests (15 tests) — necessary but keep thin

One test per major liability category. Proves the full chain works after deployment: TPP request → API Gateway → Authoriser → Lambda → Payment Hub/CDP → Response. Every E2E scenario should map directly to an AED fine. If it doesn't protect against a fine, push it down to integration or unit.

Don't build the whole pyramid at once. This order maximises liability coverage fastest:
